1. Roles & Context
1.1 Target Users
SaveTune Mobile is built for audiophiles, music enthusiasts, and power users who want true high-fidelity audio on Android — specifically those who already have a Spotify account and can supply their sp_dc session cookie. No OAuth flow, no app registration hurdles.
1.2 Core Problem
Official Spotify & streaming apps do not expose true lossless FLAC downloads to users.
Premium subscription paywalls block high-res audio on mobile.
Existing tools are desktop-only (like the Wails-based SaveTune desktop app).
Users want offline-first playback with synchronized lyrics, waveforms, and a polished dark-mode UI on Android.
1.3 Unique Constraint — No OAuth / No Login
The app does NOT use Spotify OAuth or any third-party login screen. Instead:
Users paste their sp_dc cookie (extracted from browser devtools) into the app's Settings screen.
The Go backend uses this cookie to authenticate requests to Spotify's internal APIs (same mechanism browser clients use).
This avoids Spotify rate-limiting OAuth apps and allows access to the full metadata + CDN URLs.
The sp_dc key is stored only in Android's EncryptedSharedPreferences — never sent to any external server.

2. Architecture
2.1 High-Level Topology
SaveTune Mobile uses a local-first, embedded architecture: the Go backend runs entirely on the Android device as a background service — no remote server required. The Flutter UI communicates with it via localhost HTTP.

2.2 Flutter ↔ Go Communication
The Flutter app launches the Go binary as a foreground service on app start. All communication is via localhost REST — no IPC complexity, no FFI binding needed. This also lets the Go server handle long-running downloads independently of the Flutter lifecycle.
Communication Flow

                                       │
                    ┌──────────────────┼──────────────────┐
                    ▼                  ▼                  ▼
           Spotify Internal API   CDN Audio Stream    SQLite DB
           (sp_dc cookie auth)   (FLAC/OGG mux)     (local cache)
2.3 Authentication Design (sp_dc Cookie)
No traditional auth flow. Instead:
User opens Settings → pastes sp_dc cookie value.
Flutter POSTs it to Go server: POST /api/config/spdc
Go validates the cookie by hitting Spotify's token endpoint: GET https://open.spotify.com/get_access_token
On success, Go stores the sp_dc in Android Keystore via JNI call and caches a short-lived Bearer token in memory.
All subsequent Spotify API calls use the cached Bearer token, refreshing automatically when it expires.

6. Data Flow & Security
6.1 Download Data Flow
Flutter sends POST /api/v1/download with { spotify_id, format }.
Go validates sp_dc token is still valid (refresh if needed).
Go calls Spotify internal track endpoint to get CDN stream URL (authenticated with Bearer token derived from sp_dc).
Go launches goroutine: streams CDN audio → pipes through FFmpeg → writes FLAC to /Music/SaveTune/Artist-Title.flac.
FFmpeg embeds metadata: title, artist, album, year, cover art (downloaded separately), track number.
Progress events pushed via WebSocket every 500 ms.
On completion, Go inserts track record into SQLite tracks table.
Flutter receives WebSocket "complete" event, refreshes library screen.
6.2 Input Validation & Sanitization
Go backend: all search query strings are URL-encoded before forwarding to Spotify.
spotify_id parameters are validated against regex ^[a-zA-Z0-9]{22}$ before any API calls.
Filename sanitization: sanitizeFilename() strips / \ : * ? " < > | and limits to 200 chars.
sp_dc cookie: validated by actually calling Spotify's token endpoint — no regex trust.
FFmpeg subprocess: all arguments passed as []string (never shell-interpolated) to exec.Command.

6.4 Network Security
All external calls (Spotify APIs) are HTTPS only — TLS verified.
Local Go server only binds to 127.0.0.1 (loopback) — not accessible on the network.
No analytics, no telemetry, no external logging. Fully local operation.

8.3 Key Code Skeleton — Go Main
// main.go
package main
import (
    "github.com/gin-gonic/gin"
    "savetune/db"
    "savetune/handlers"
    "savetune/spotify"
)
func main() {
    database := db.Init("savetune.db")
    spotifyClient := spotify.NewClient()
    r := gin.Default()
    r.Use(corsMiddleware()) // Allow localhost Flutter
    v1 := r.Group("/api/v1")
    handlers.RegisterConfig(v1, spotifyClient)
    handlers.RegisterSearch(v1, spotifyClient)
    handlers.RegisterDownload(v1, spotifyClient, database)
    handlers.RegisterLibrary(v1, database)
    handlers.RegisterLyrics(v1, spotifyClient, database)
    handlers.RegisterWebSocket(v1)
    r.Run("127.0.0.1:7799") // Loopback only
}
8.4 Key Code Skeleton — Flutter API Client
// lib/shared/api/client.dart
class SaveTuneApi {
  static const _base = "http://127.0.0.1:7799/api/v1";
  final _dio = Dio(BaseOptions(baseUrl: _base));
  Future<List<TrackModel>> search(String q) async {
    final res = await _dio.get("/search", queryParameters: {"q": q});
    return (res.data["tracks"] as List).map(TrackModel.fromJson).toList();
  }
  Future<DownloadJob> queueDownload(String spotifyId) async {
    final res = await _dio.post("/download", data: {"spotify_id": spotifyId, "format": "flac"});
    return DownloadJob.fromJson(res.data);
  }
}
8.5 Key Code Skeleton — Go Download Handler
// handlers/download.go
func QueueDownload(c *gin.Context) {
    var req struct { SpotifyID string `json:"spotify_id"` }
    if err := c.ShouldBindJSON(&req); err != nil { c.JSON(400, gin.H{"error": err.Error()}); return }
    // Validate spotify_id format
    if !spotifyIDRegex.MatchString(req.SpotifyID) {
        c.JSON(400, gin.H{"error": "invalid spotify_id", "code": "INVALID_ID"}); return
    }
    jobID := uuid.NewString()
    go downloader.Start(jobID, req.SpotifyID, spotifyClient, db, progressHub)
    c.JSON(202, gin.H{"job_id": jobID, "status": "queued"})
}

   cd savetune-go

   cd savetune_mobile && flutter run --release
